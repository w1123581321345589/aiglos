Intermediate 15 min Policy

Lockdown Policy

Lockdown is the highest-security policy tier. Every tool call is denied by default. The human operator must explicitly grant each tool before the agent can use it. This tutorial covers activation, grant management, and the compliance artifact lockdown produces.

Activate lockdown policy

Python
import aiglos

guard = aiglos.OpenClawGuard(
    agent_name="critical-agent",
    policy="lockdown",      # deny everything by default
    log_path="./agent.log",
)

# Before any tool runs, you must grant it explicitly
guard.allow_tool("filesystem.read_file", reason="Agent needs to read src/ files")
guard.allow_tool("web.search", reason="Agent needs web search for research")

Granting tools via CLI

Shell
# List what the agent is trying to use (blocked calls)
aiglos policy lockdown --pending

# Grant a specific tool
aiglos policy allow --tool filesystem.read_file --reason "Code review agent"

# View current grants
aiglos policy grants

# Revoke a tool
aiglos policy revoke --tool shell.execute

Using the permission recommender first

If you have existing session history, let Aiglos recommend the minimum viable allowlist before activating lockdown:

Shell
# What did this agent actually use in the last 30 days?
aiglos policy recommend --agent my-agent

# Output:
# Recommended allowlist (from 847 sessions):
#   filesystem.read_file  — 823 sessions
#   web.search            — 412 sessions
#   shell.execute         — 3 sessions (unusual — review before granting)

# Apply the recommendation as a lockdown starting allowlist
aiglos policy recommend --agent my-agent --apply
Lockdown produces a grant log. Every allow_tool() call is recorded with timestamp, reason, and operator identity. This log is included in the signed session artifact — exactly the evidence NDAA §1513 compliance reviews require.

Policy tiers reference

Shell
# permissive: WARN only, nothing blocked
# enterprise: BLOCK at score ≥ 0.75 (default)
# strict:     BLOCK at score ≥ 0.60
# federal:    BLOCK at score ≥ 0.50, mandatory HITL
# lockdown:   BLOCK everything not in allowlist

aiglos attach --policy lockdown

Advanced 25 min Benchmarking

Run GOVBENCH

GOVBENCH is the first governance benchmark for AI agents. It measures five dimensions of failure resistance across 20 attack scenarios in under 60 seconds. Run it before a major deployment, after a policy change, or weekly as a regression check.

Basic run

Shell
pip install aiglos

# Run all 5 dimensions (20 scenarios, ~45 seconds)
aiglos benchmark run

# Single dimension
aiglos benchmark run --dimension D1

# JSON output for CI integration
aiglos benchmark run --format json | tee govbench-$(date +%Y%m%d).json

Understanding your grade

A≥ 90
B≥ 75
C≥ 60
D≥ 45
F< 45

Aiglos enterprise policy typically scores B (≈87). The primary gaps are in D4 (RL feedback loop resistance) where full detection requires access to the reward signal mechanism. Target B or above for production deployments in regulated industries.

Intermediate 10 min Security Audit

Scan Exposed Instances

aiglos scan-exposed probes an OpenClaw gateway for the attack surface documented in GHSA-g8p2-7wf7-98mq and CVE-2026-25253 — the vulnerability that left 40,214 instances exposed. Run it against your own deployment or any gateway you're authorized to test.

Only run against gateways you own or are authorized to test. This command makes active HTTP/WebSocket requests to the target.

Basic scan

Shell
# Scan your local OpenClaw instance
aiglos scan-exposed --target http://localhost:3000

# Scan a remote gateway (must be authorized)
aiglos scan-exposed --target https://your-openclaw.example.com

# JSON output
aiglos scan-exposed --target http://localhost:3000 --format json

What it checks

Output
Aiglos scan-exposed — http://localhost:3000
───────────────────────────────────────────

  ✓ Gateway reachable (HTTP 200)
  ✓ WebSocket endpoint present

  ✗ CRITICAL: Gateway auth not required (allow_remote=true, no token)
    → CVE-2026-25253 / GHSA-g8p2-7wf7-98mq attack surface confirmed
    → T68 INSECURE_DEFAULT_CONFIG

Phase 3: Configuration Exposure
  ✗ HIGH: Config endpoint accessible without auth
  ✓ Admin UI requires auth

Phase 4: WebSocket Handshake
  ✗ CRITICAL: Auth token exfiltration vector confirmed
    → Matches GHSA-g8p2-7wf7-98mq attack chain

───────────────────────────────────────────
Grade: F (2 critical, 1 high, 0 medium)

Remediation:
  1. Set gateway.auth.mode to "token" with random 32-byte value
  2. Set gateway.bind to "loopback"
  3. Run: aiglos audit --hardening-check

Docs: docs.openclaw.ai/gateway/security#hardened-baseline

Using in CI to prevent accidental exposure

YAML — .github/workflows/exposure-check.yml
name: OpenClaw Exposure Check
on: [push, pull_request]

jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install aiglos
      - name: Check for exposed config
        run: |
          aiglos scan-exposed \
            --target http://localhost:3000 \
            --fail-on-critical \
            --format json | tee scan-results.json

The 40,214 number

SecurityScorecard published in February 2026 that 40,214 OpenClaw gateway instances were exposed to the public internet with no authentication — the default allow_remote=true configuration. CVE-2026-25253 was the mechanism. T68 INSECURE_DEFAULT_CONFIG catches it at the config layer. aiglos scan-exposed confirms it at the network layer.
