What was built and why.
T79 PERSISTENT_MEMORY_INJECT (score 0.92, critical) is the 79th rule and the most novel detection class in the taxonomy. The distinction from T31 is precise: T31 fires on in-session memory writes with injection content, cleared on session close. T79 fires specifically on writes to persistent backends that survive session close and get injected into every future session — Gigabrain SQLite databases, Chroma, Pinecone, Qdrant, Weaviate, mem0, Letta, Zep. The adversary compromises the memory once and gets permanent influence over future behavior without needing to re-inject. This is categorically worse, which is why the score (0.92) sits above T31. Both conditions required — path matches a known memory backend AND content contains injection patterns — for precision over recall. Read operations early-exit cleanly.
aiglos/integrations/gigabrain.py — 200 lines, zero dependencies. declare_memory_backend() takes a backend name and optional custom path, registers all known paths for that backend with the T79 enforcement engine, and records the registration on the guard. gigabrain_autodetect() is the zero-config path — scans every known Gigabrain path, registers them all regardless of whether a database exists yet (because T79 should fire on creation too), and returns a session. is_registered_memory_path() is the bridge back to T79 matching — any path in the global registry gets flagged even if the tool name is unexpected. Covers 8 backends beyond Gigabrain.
GIGABRAIN_MEMORY_POISON campaign pattern (confidence 0.95) — T31 followed by T79. The two-step sequence: the adversary first tests in-session injection (T31) to verify it fires correctly, then commits the same content to the persistent store (T79). The T79 amplifier at 1.5x means any exfil that follows in the same session gets elevated scoring. This is the only campaign pattern that specifically targets the two-tier memory architecture (temporary + persistent) that Gigabrain and similar tools create.
