This is the right question and the answer requires rethinking the fundamental architecture of how Aiglos reaches customers. Let me think through this from first principles.

**The core problem with the current architecture**

Right now Aiglos is positioned as a proxy server you deploy in front of your MCP stack. That means: provision infrastructure, open ports, configure yaml, redirect traffic. Even for a sophisticated team that’s a day. For enterprise procurement it’s a quarter. That’s the wrong deployment model for blitz scale.

The parallel: Stripe vs. traditional payment processors. Traditional processors required integration teams, bank agreements, and weeks of certification. Stripe made it one API key and ten lines of code. Revenue in the time it took to read the docs. Aiglos needs to make the same move.

-----

**The Embedded Library Architecture**

The shift is from proxy (infrastructure you run) to library (code you import). This is the architectural decision that unlocks seconds-to-deploy.

Instead of pointing your MCP client at a proxy server, you do this:

```python
import aiglos  # one line, monkey-patches mcp.client.*
```

That’s it. Aiglos installs itself in-process, intercepts every MCP call before it leaves your runtime, and routes telemetry to the Aiglos cloud. No server. No ports. No yaml. No ops.

The same move works in TypeScript:

```typescript
import 'aiglos/auto'  // auto-registers before any MCP calls
```

The embedded model is actually more powerful than the proxy model for detection purposes. In-process means Aiglos sees the full Python call stack, the agent’s goal state, the arguments before serialization. The proxy sees wire-format JSON. Richer data, zero deployment friction.

**The Four-Tier Deployment Ladder**

The architecture supports customers entering at any level and moving up:

1. **Library embed** (seconds) - pip install, one import, usage-based billing starts immediately. Target: individual developers, data engineering teams.
1. **Cloud proxy** (minutes) - `docker run aiglos/aiglos` with an API key routes traffic through Aiglos’s managed infrastructure. Target: teams that want the proxy model without running infrastructure.
1. **On-prem deployment** (hours) - customer runs the Aiglos container in their own cloud. Flat-rate enterprise contract. Target: enterprise security teams, air-gap-capable deployments.
1. **Program-level contract** (the DoD channel) - custom integration, C3PAO evidence package, performance guarantee. Revenue is $5-20M TCV per contract. Target: DoD-adjacent prime contractors.

The critical insight is that tiers 1-3 are self-serve. A developer discovers Aiglos via PyPI or the MCP registry, activates it in minutes, their enterprise upgrades them to tier 3, and the tier 3 relationship is the on-ramp to tier 4. The DoD channel does not require cold outreach; it requires tier 3 enterprise customers who are already defense-adjacent.
