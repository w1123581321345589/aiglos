## Problem 1: Injection Scanner False Positives ‚Äî CORRECTNESS BUG

**What‚Äôs wrong.** The phrase corpus uses plain substring matching. ‚ÄúYour task is now‚Äù fires on ‚Äúyour task is nowhere near complete.‚Äù ‚ÄúFirst do‚Äù fires on ‚Äúfirst do the setup then run the tests.‚Äù ‚ÄúBefore doing anything‚Äù fires on ‚Äúbefore doing anything else, check the logs.‚Äù All five of these are confirmed false positives that would produce MEDIUM verdicts on completely clean tool outputs.

**Fix.** Replace substring matching with word-boundary-aware matching using `\b` regex anchors, and add a minimum-word-count requirement: the surrounding phrase must contain at least one more word that reinforces an instructional context (a verb, an object, a redirect target). For the most ambiguous phrases (‚Äúfirst do‚Äù, ‚Äúinstead do‚Äù), require a second signal before scoring them ‚Äî they should contribute 0.10 to score, not 0.50, unless paired with another hit.

## Problem 2: Intent Predictor Uses a Single Global Model File ‚Äî CORRECTNESS BUG

**What‚Äôs wrong.** `DEFAULT_MODEL_PATH = Path.home() / ".aiglos" / "intent_model.json"` is a single shared file for all agents on the machine. A deployment running a financial agent (T37-heavy sequence profile) and a code agent (T07/T19-heavy) will have their models overwrite each other on every retrain. The predictions for the financial agent will be polluted with code agent patterns and vice versa.

**Fix.** Key the model file by `agent_name`: `~/.aiglos/models/{agent_name}.json`. `IntentPredictor.__init__` derives the path from the agent name passed in, and `enable_intent_prediction()` in `OpenClawGuard` passes `self.agent_name`. One model per agent profile, not one global model.

## Problem 3: Forecast Elevations Never Decay ‚Äî DESIGN BUG

**What‚Äôs wrong.** Once `_active_adjustments` contains a rule, it stays there for the entire session with no decay or re-evaluation. If T37 is elevated to Tier 3 at step 5 because of a 73% prediction, but by step 40 the agent has run 35 completely clean file operations with zero T37 signal, the elevation stays. This produces Tier 3 friction for the entire remaining session with no evidence it‚Äôs still warranted.

**Fix.** Add a `_elevation_step` to each `ForecastAdjustment`. In `after_action()`, re-check active elevations after each new prediction: if the probability of the elevated rule has dropped below half its triggering threshold for three consecutive steps, remove the elevation. Add a `steps_at_elevation` counter and log when elevations are applied, extended, or released ‚Äî this data is critical for tuning.

## Problem 4: CausalTracer Attribution Window is Step-Count-Based, Not Context-Aware ‚Äî DESIGN ISSUE

**What‚Äôs wrong.** `_ATTRIBUTION_WINDOW = 15` steps is hardcoded. For a coding agent that does 200 tool calls per session (common with Cursor or Claude Code), 15 steps is less than 10% of the session ‚Äî most injections that happen in the first third of a long session will get LOW or NONE attribution even if the causal link is real. For a simple agent that does 8 total calls, 15 steps covers the entire session.

**Fix.** Make the attribution window percentage-based: `min(15, max(8, session_length * 0.20))` computed at attribution time from the total action count. Expose it as a configurable parameter (`attribution_window_pct`, default 0.20). Document the tradeoff: larger windows produce more attributions but also more false positives where an injection happened to be in context but didn‚Äôt cause the action.

## Problem 5: Multi-Agent Registry Has No Parent Verification ‚Äî SECURITY BUG

**What‚Äôs wrong.** `register_spawn(parent_id, child_id, ...)` trusts the caller-supplied `parent_id` without checking that a session with that ID actually exists in the registry. An attacker who can inject a spawn call can claim any arbitrary session as parent, manufacturing a false provenance chain in the audit artifact. The registry is supposed to be the proof of lineage; if it doesn‚Äôt verify lineage, it‚Äôs just a log.

**Fix.** In `register_spawn()`, check that `parent_id` matches either `self.root_session_id` or an existing `child_id` in `_children`. If it doesn‚Äôt, raise a `RegistryIntegrityError` with a log entry. Add a corresponding test: attempt to register a child with a nonexistent parent, verify the error fires.

## Problem 6: SessionArtifact.extra is an Untyped Dict ‚Äî MAINTAINABILITY ISSUE

**What‚Äôs wrong.** Three separate sections (injection, causal, forecast) are merged into `artifact.extra` via dict `.update()`. There‚Äôs no schema, no validation, and no guarantee the sections don‚Äôt overwrite each other if key names collide. Currently `injection_summary`, `injection_flagged`, `causal_attribution`, `forecast_summary`, `forecast_adjustments`, `forecast_snapshots` are the keys ‚Äî but this is implicit, not enforced.

**Fix.** Add an `ArtifactExtensions` dataclass with typed optional fields for each section: `injection: Optional[dict]`, `causal: Optional[dict]`, `forecast: Optional[dict]`. Replace the three `artifact.extra.update()` calls with `artifact.extensions.injection = ...` etc. Add `.extra` as a property that merges them for backward compat. This makes the structure inspectable, testable, and prevents key collisions.

## Problem 8: ObservationGraph Has No Thread Safety for Concurrent Agents ‚Äî ROBUSTNESS ISSUE

**What‚Äôs wrong.** `_conn()` opens a new connection per call with WAL mode. WAL is good for concurrent readers but still has write serialization. If two agents on the same machine both call `ingest()` simultaneously, you get intermittent write conflicts and possible data loss under Python‚Äôs GIL. There‚Äôs no lock around write operations.

**Fix.** Add a `threading.Lock` at the `ObservationGraph` instance level, acquired for write operations (`ingest`, `ingest_causal_result`, etc.) but not read operations. Document that multi-process deployments should use separate `db_path` values per process. This is a narrow fix ‚Äî the bigger solution (connection pool) is over-engineering for the current scale.

## Problem 9: Version Has Two Sources of Truth ‚Äî MAINTAINABILITY ISSUE

**What‚Äôs wrong.** `__version__` is hardcoded as a string in `aiglos/__init__.py` at line 41. It‚Äôs also in `pyproject.toml` as `version = "0.1.0"` (which is stale ‚Äî never updated). Every version bump requires two changes and the two will eventually diverge. Tests assert on the string literal, which means a missed update produces silent failures.

**Fix.** Make `pyproject.toml` the single source of truth. Use `importlib.metadata.version("aiglos")` in `__init__.py` to read it dynamically, with a fallback to the hardcoded string for development installs. Tests then import `aiglos.__version__` as before, but the actual value comes from one place. Update `pyproject.toml` to `version = "0.10.0"` now.
