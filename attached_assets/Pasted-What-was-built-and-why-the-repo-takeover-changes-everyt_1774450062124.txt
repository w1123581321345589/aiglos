What was built and why the repo takeover changes everything.
T81 PTH_FILE_INJECT (score 0.98, critical) is the rule that was missing until yesterday. The full attack chain is now covered: T30 fires when a compromised package installs. T81 fires the moment litellm_init.pth lands in site-packages — before the first Python restart, before the .pth executes, before a single credential leaves the machine. Score 0.98 because writing a .pth file to site-packages is never legitimate agent behavior. The known-malicious filename set (litellm_init.pth) fires at score 1.0 regardless of content — the name alone is the signal.
REPO_TAKEOVER_CHAIN campaign (confidence 0.97) — the pattern the incident demonstrates. T30 (install) → T81 .pth write → T04 credential harvest → T41 exfil → T30 again, the attacker using the stolen GitHub/PyPI token to publish a new poisoned release or close the security issue. The final T30 is what makes this a repo takeover rather than a one-time attack. The attacker is now the maintainer. Every future pip install litellm is a potential reinfection. The amplifiers — T81 at 2.0x, T04 at 1.5x — reflect that .pth + credential harvest in the same session is near-certain confirmation.
aiglos scan-deps — 300 lines. Checks: installed packages against COMPROMISED_PACKAGES (litellm 1.82.7 and 1.82.8 as first entries), all site-packages directories for .pth files with suspicious content or known-malicious names, ~/.config/sysmon/ persistence directory, uv cache for compromised wheels, and transitive exposure across dspy, smolagents, LangChain, LangGraph, CrewAI. Returns a ScanResult with risk level, findings, and concrete remediation steps. Exits with non-zero status if compromised — pipeable into CI.
The repo takeover consideration. The original tweet described a poisoned PyPI release. The GitHub issue being closed “not planned” by the owner means the attacker has the maintainer’s credentials and is actively suppressing detection. This changes the threat model: it’s no longer “patch to a safe version” — it’s “treat the entire repository as compromised until ownership is demonstrably restored.” scan-deps reflects this in its reporting: it flags not just the known compromised versions but also warns that ALL litellm versions should be treated as suspect. The COMPROMISED_PACKAGES dict is designed to be updated — it’s the living registry that aiglos autoresearch watch-ghsa feeds into as new confirmed-bad versions are identified.
