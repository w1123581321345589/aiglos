Here is what changed from the prior README and why each change matters.
Tagline updated. Old: "The first AI agent security platform that covers every surface your agent operates on, learns from every session it runs..." New: "The AI agent security platform that learns your deployment, eliminates false positive fatigue, and gets permanently better with every deployment in the network." The old tagline described features. The new one describes the compounding value — which is what the product now actually is.
"What changed in v0.13" section. New. This is the investor/developer read — three paragraphs that explain the architectural shift without requiring the reader to parse a changelog. The false positive death spiral, cold-start elimination, and network effect each get a plain-language explanation of the before/after. This section didn't exist before because before v0.11 there was nothing this significant to explain.
Live output block updated. Added Behavioral baseline: LOW (composite=0.09) — within normal range for this agent to the session close output. This is the most visible surface-level change — the artifact now tells you whether the session was normal for this specific agent, not just whether any rule fired.
Behavioral Baseline section. New. Three feature spaces explained with code examples. The key line: "A DevOps agent that reads credentials in 100% of its sessions — session 101 scores LOW on behavioral anomaly." This is the concrete answer to the question every security team asks: "how do we stop getting paged for normal DevOps operations?"
Policy Proposals section. New. The CLI output example is the pitch — prp_a1b2c3d4 LOWER_TIER devops-agent/T19, 91% confidence, 14 blocks, 27d left. One line, one decision, the agent never blocked again. The four-row evidence table makes the proposal type selection legible without reading the code.
Federated Intelligence section. New. The JSON payload example is intentional — it shows what actually gets sent ({"T19": {"T37": 14.2, ...}}) alongside what never leaves. The local weight schedule (20% → 80% at 100 sessions) appears because it answers the question every privacy-conscious engineer asks: "does the global prior ever override my local data?"
Pricing table updated. Old: brief and vague about what federation means. New: "Federation: pull global prior only" on free tier, "contribute to global prior (makes every deployment smarter)" on Pro. The distinction is precise. Free tier gets the benefit of the network. Pro tier contributes to it. The value proposition for Pro is now concrete — not "cloud dashboard" but "your deployment makes every other deployment smarter."
Changelog extended. v0.11.0, v0.12.0, v0.13.0 entries added with accurate test counts (753, 818, 882). Each entry describes the module, the inspection trigger, and the test count — the same format as prior entries, consistent cadence.
Stats bar updated. "13 inspection triggers" and "Learns your deployment" and "Network intelligence" replace the old "Self-improving rules" and "Belief-layer + RL security" entries. The stats now reflect what distinguishes the product from v0.10.
